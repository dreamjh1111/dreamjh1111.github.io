# MySQL Aurora PostgreSQL Comparison Article Design

## Goal

`MySQL(InnoDB)`, `Aurora MySQL-compatible`, `PostgreSQL`의 차이를 입문자와 실무 개발자 사이 난이도로 깊이 있게 비교하는 글을 작성한다. 단순 기능 나열이 아니라, 각 제품의 탄생 배경과 설계 철학, 스토리지 구조, 트랜잭션/MVCC, 락, 인덱스, 복제/운영 방식 차이를 연결해 설명하고, 각 파트 끝마다 사용자 입장에서 어떤 선택 포인트가 생기는지 정리한다.

## Audience

- `InnoDB` 시리즈를 읽고 "그럼 DB 제품별 차이는 뭘까?"가 궁금해진 독자
- 백엔드 입문자이지만 너무 얕은 비교표는 아쉬운 독자
- 실무에서 `MySQL`, `Aurora`, `PostgreSQL` 선택 이유를 설명해야 하는 개발자

## Structure

1. 도입: 왜 엔진 내부를 공부한 뒤 제품 비교까지 가야 하는가
2. 제품별 탄생 배경과 설계 철학
3. 스토리지 구조 비교
4. 트랜잭션, MVCC, 격리 수준 비교
5. 락 모델 비교
6. 인덱스와 쿼리 최적화 관점 비교
7. 복제, 고가용성, 운영 관점 비교
8. 실무 선택 기준
9. 마무리와 참고 자료

## Comparison Axes

- 탄생 배경
- 설계 철학
- 스토리지 구조
- 트랜잭션과 MVCC
- 기본 격리 수준과 락 모델
- 인덱스와 확장 기능
- 복제/고가용성/장애 복구
- 개발자와 운영자 입장에서 체감되는 차이

## Content Notes

- `Aurora`는 이번 글에서 `Aurora MySQL-compatible`로 한정한다.
- 역사/배경 설명은 공식 역사 문서나 공식 블로그를 바탕으로 하되, 철학 해석은 추론임을 표시한다.
- 기술 비교는 가능한 한 공식 문서와 제품 문서 기준으로 서술한다.
- 각 파트 끝에 반드시 `그래서 사용자 입장에서는?` 소제목을 둔다.
- 너무 많은 비교 축을 한 표에 몰지 않고, 핵심 축별로 표와 설명을 함께 쓴다.
